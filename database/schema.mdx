## Overview

GAM uses a single MySQL database named `db_gam`. The schema covers academic organization (divisions, careers, groups), people and users with role-based access, physical infrastructure (buildings and rooms), furniture inventory, reservations, loans, damage reports, and an activity log (bitácora).

<Note>
All `id_estatus` columns use a soft-delete pattern. Records are never physically deleted — they are marked `'Eliminado'` instead.
</Note>

### Database initialization

```sql
DROP DATABASE IF EXISTS db_gam;

CREATE DATABASE IF NOT EXISTS db_gam;

USE db_gam;
```

---

## Connection configuration

The Java back-end connects via `ConexionMySQL.java` using the MySQL Connector/J driver:

```java
String user     = "root";
String password = "Melapelas2";
String url      = "jdbc:mysql://localhost:3306/db_gam"
                + "?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=UTC";

Class.forName("com.mysql.cj.jdbc.Driver");
conn = DriverManager.getConnection(url, user, password);
```

<Warning>
The credentials above are the development defaults committed to source control. Change `user` and `password` before deploying to any non-local environment.
</Warning>

| Parameter | Value |
|-----------|-------|
| Host | `localhost` |
| Port | `3306` |
| Database | `db_gam` |
| Driver | `com.mysql.cj.jdbc.Driver` |
| SSL | `false` |
| Timezone | `UTC` |

## Tables

The database contains **13 tables**:

| Table | Purpose |
|-------|---------|
| `tbl_div_academica` | Academic divisions (e.g. Tecnologías de la Información) |
| `tbl_carreras` | Degree programs belonging to a division |
| `tbl_grupos` | Student groups / cohorts within a career |
| `tbl_personas` | Physical persons — shared by users, students, and administrators |
| `tbl_administrativos` | Administrative staff linked to a person and division |
| `tbl_alumnos` | Students linked to a person and group |
| `tbl_usuarios` | System accounts with role, shift, and status |
| `tbl_edificios` | Campus buildings belonging to a division |
| `tbl_salones` | Rooms (classrooms, labs, halls) inside buildings |
| `tbl_inventario` | Individual furniture/equipment items assigned to rooms |
| `tbl_reportes_mobiliario` | Damage or incident reports raised against inventory items |
| `tbl_historial_reporte` | Audit trail of every status change on a report |
| `tbl_reservaciones` | Room reservations made by users |
| `tbl_prestamos_mobiliario` | Equipment loans checked out to users |
| `tbl_bitacora` | System-wide activity log populated by triggers |

  <Accordion title="tbl_usuarios — System users">

  Authentication and authorization table. Passwords are stored as SHA-256 hashes.

  | Column | Type | Constraints |
  |--------|------|-------------|
  | `id_usuario` | `INT` | `NOT NULL`, `AUTO_INCREMENT`, `PRIMARY KEY` |
  | `id_persona` | `INT` | `NOT NULL`, `FK → tbl_personas` (CASCADE) |
  | `matricula` | `VARCHAR(50)` | `NOT NULL`, `UNIQUE` |
  | `password_hash` | `CHAR(64)` | `NOT NULL` — SHA-256 hex digest |
  | `id_rol` | `ENUM('Administrador','Coordinador','Docente','Alumno')` | `NOT NULL` |
  | `id_turno` | `ENUM('Matutino','Vespertino','Despresurizado')` | `NOT NULL` |
  | `id_estatus` | `ENUM('Activo','Inactivo','Eliminado')` | `NOT NULL` |

  ```sql
  CREATE TABLE tbl_usuarios(
      id_usuario INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
      id_persona INT NOT NULL,
      matricula VARCHAR(50) NOT NULL UNIQUE,
      password_hash CHAR(64) NOT NULL,
      id_rol ENUM(
          'Administrador',
          'Coordinador',
          'Docente',
          'Alumno'
      ) NOT NULL,
      id_turno ENUM(
          'Matutino',
          'Vespertino',
          'Despresurizado'
      ) NOT NULL,
      id_estatus ENUM(
          'Activo',
          'Inactivo',
          'Eliminado'
      ) NOT NULL,
      FOREIGN KEY (id_persona) REFERENCES tbl_personas(id_persona) ON DELETE CASCADE ON UPDATE CASCADE
  );
  ```

  <Tip>
  Passwords are hashed with `SHA2(password, 256)` at insert time. The `sp_login_usuario` procedure re-hashes the supplied plain-text password and compares it against `password_hash`.
  </Tip>

  |--------|------|-------------|
  | `id_inventario` | `INT` | `NOT NULL`, `AUTO_INCREMENT`, `PRIMARY KEY` |

  ```sql
      id_edificio INT NOT NULL,
      id_ubicacion ENUM('Edificio','Administracion'),
      id_tipo_mobi ENUM(
          'Silla',
          'Mesa',
          'Pizarron',
          'Ventana',
          'Cañon',
          'Pantalla',
          'Extension',
          'HDMI',
          'Proyector'
      ) NOT NULL,
      id_categoria_mobi ENUM(
          'Tecnología',
          'Cables',
          'Eléctrico',
          'Papelería'
      ) NOT NULL,
      id_estatus ENUM(
          'Activo',
          'Inactivo',
          'Eliminado'
      ) NOT NULL,
      id_estatus_actual ENUM(
          'Disponible',
          'En uso'
      ) NOT NULL,

      FOREIGN KEY (id_edificio) REFERENCES tbl_edificios(id_edificio) ON DELETE CASCADE ON UPDATE CASCADE
  ```

  <Info>
  `id_estatus` tracks the item's overall lifecycle (active, inactive, or deleted). `id_estatus_actual` tracks its moment-to-moment operational state (`Disponible` / `En uso`). The trigger `tg_actualizar_inventario_salida` automatically sets `id_estatus_actual = 'En Uso'` when a loan is created.
  </Info>

## Entity relationships

The diagram below summarizes the foreign key relationships between the core tables.

```

    ├── tbl_carreras
    │       └── tbl_grupos
    │               └── tbl_alumnos ──── tbl_personas
    └── tbl_edificios
            └── tbl_salones
                    ├── tbl_inventario
                    │       ├── tbl_reportes_mobiliario ── tbl_historial_reporte
                    │       └── tbl_prestamos_mobiliario
                    └── tbl_reservaciones

tbl_personas
    ├── tbl_administrativos
    └── tbl_usuarios
            ├── tbl_reportes_mobiliario
            ├── tbl_historial_reporte
            ├── tbl_reservaciones
            └── tbl_prestamos_mobiliario
```
