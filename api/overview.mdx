# API Overview

The GAM backend exposes a JAX-RS REST API deployed on a Java EE application server (GlassFish / Payara). All resources are mounted under the base path `/IntegradoraII_Web/webresources/`.

## Base URL

```
http://<host>:<port>/IntegradoraII_Web/webresources/
```

The web front-end calls this URL directly. The mobile app proxies all `/api/*` requests through `proxy.conf.json` to the same origin during development — the path prefix `/api` is stripped by the proxy and never reaches the Java application.

## Authentication

GAM uses **session-based authentication**. There is no token header — the browser session established after a successful login is used for subsequent requests.

**Login flow:**

1. `POST /usuario/login` with a JSON body containing `matricula` and `contrasenia`.
2. On success the server returns the full user object (`200 OK`). Store relevant fields (e.g. `id_usuario`, `id_rol`) in `localStorage` for client-side use.
3. On failure the server returns `401 Unauthorized` with `{"error": "Credenciales inválidas o usuario inactivo."}` .

```bash
curl -s -X POST \
  http://localhost:8080/IntegradoraII_Web/webresources/usuario/login \
  -H "Content-Type: application/json" \
  -d '{"matricula": "A12345", "contrasenia": "secreto"}'
```

## Content Types

| Scenario | Request `Content-Type` |
|---|---|
| Most endpoints (JSON body) | `application/json` |
| `/usuario/eliminar` | `application/x-www-form-urlencoded` |
| `/reservacion/insertar`, `/reservacion/actualizar`, `/reservacion/eliminar` | `application/x-www-form-urlencoded` |
| `/reporte/insertar`, `/reporte/eliminar` | `application/x-www-form-urlencoded` |

All responses are `application/json`.

## Error Format

Errors are returned as a JSON object with an `error` key:

```json
{"error": "Descripción del error"}
```

Some endpoints return a `mensaje` key for success confirmations:

```json
{"mensaje": "Operación completada correctamente."}
```

<Note>
  HTTP status codes are not always semantically strict in this API. Some error conditions (particularly in `reservacion` endpoints) return `200 OK` with an `error` field in the body. Always inspect the response body.
</Note>
